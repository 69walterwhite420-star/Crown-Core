# Crown — Спецификация ядра

**v6.0 · Сплиттер + индексатор · Больше ничего**

Фабрика, игры, централизованный слой и фронт — в `post-core.md`. Verifier — публичный крейт `reduce`. Признание — пиннутый адрес, не свойство событий.

---

## 1. Ядро — это два артефакта

| Артефакт | Где | Держит деньги | Держит ключи | Изменяем |
|---|---|---|---|---|
| `splitter` | Solana | **нет, никогда** | нет | нет |
| `crown-index` | канистра ICP | нет | нет | нет (после S4) |

Между ними нет канала. Сплиттер не знает об индексаторе. Индексатор читает сплиттер из публичного чейна и больше ничего от него не хочет.

```
Донор ──транзакция──► splitter ──► получатель   (gross целиком)

                             emit Settled(donor, recipient, gross)
                                        │
                 SOL RPC-канистра (NNS) │ читает, консенсус ≥3 провайдеров
                                        ▼
                                  crown-index
                                  reduce → книга
                                        │
                                  query │ бесплатно, permissionless, certified
                                        ▼
                                  кто угодно
```

## 2. Закон репутации

Книга — отображение `(chain, donor, recipient) → gross`, где `gross` — сумма USDC в minor units (6 знаков), `u128`.

```
reduce(book, Settled { chain, donor, recipient, gross, .. })
    = book.insert((chain, donor, recipient), book.get(...) .checked_add(gross)?)
```

Всё. Одно правило, одна ветка.

**Свойства, которые из него следуют и обязаны быть доказаны тестами:**

- **Монотонность.** Значение только растёт. Нет операции, уменьшающей баланс.
- **Аддитивность / нечувствительность к порядку.** `fold` над любой перестановкой множества сеттлментов даёт ту же книгу. Кросс-чейн упорядочивание не нужно и не существует.
- **Идемпотентность на уровне ингеста.** Один сеттлмент не начисляется дважды. Обеспечивается курсором (§5), не памятью о виденном.
- **Пересчёт.** `book[(c,p,s)] == Σ gross` по всем `Settled` этой тройки. Тривиально проверяемо.

**Начисляется донору**, то есть тому, с чьего кошелька реально утянули токены. Не тому, кто вызвал. Это обеспечено контрактом (§3), а не параметром события.

**Ключ включает `chain` — идентификатор кластера.** Devnet и mainnet — разные вселенные с разными книгами; записи между ними не смешиваются и не переносятся. Связывание личностей поверх кошельков — задача отображения, оно никогда не питает книгу (см. `post-core.md`).

**Возврата нет, дельт нет, decay нет.** Репутация тождественно равна сумме честно уплаченного. Она непередаваема, непродаваема, локальна для получателя. Потеря ключей = потеря репутации; это свойство непередаваемости, не баг.

## 3. Сплиттер: три гарантии, ноль баланса

Одна immutable программа. Не фабрика, не шаблон, не инстанс — **один адрес**.

Единственная инструкция: `donate(gross)`; получатель — аккаунт транзакции.

```
transfer_checked(ATA донора → ATA получателя, gross)      // целиком
emit Settled(donor = подписант, recipient, gross)        // event-CPI
```

**Гарантия 1 — `gross` = реально сдвинутые деньги.** Не декларация. Если перевод не прошёл, транзакция ревертится, события нет.

**Гарантия 2 — донор реален.** `donor` не параметр. Токены тянутся с ATA, чей authority подписал транзакцию. Подарить репутацию чужому адресу невозможно, даже за реальные деньги.

**Гарантия 3 — программа никогда не имеет баланса.** Один прямой перевод донор→получатель. Программа не является владельцем средств ни в один момент. Это не «она не крадёт» — **красть нечего**. Non-custodial по построению, а не по логике.

**Комиссии нет.** Получатель получает `gross` целиком; репутация тождественна дошедшему до него доллару без поправок. Комиссии игр (post-core) снимаются эскроу **до** вызова сплиттера и через него не проходят: в `gross` их не бывает по построению.

**Нулевой донат ревертится.** `require(gross > 0)`: пустой перевод не рождает события.

**Неизменяемость — часть спеки, не операционка.** После деплоя `solana program set-upgrade-authority --final`. DoD проверяет, что `solana program show` печатает `Authority: none`. Единственный параметр деплоя — минт USDC; он константа программы.

Программа крошечная (цель: < 100 строк логики) именно потому, что она единственное место, где может быть настоящая дыра.

## 4. Признание: пиннутый адрес

Индексатор засчитывает `Settled` **только от адреса сплиттера, взятого из `config/`**. Больше ни от кого.

Почему не «любая программа с `Settled`»: злоумышленник эмитит `gross = 1e18` со своей программы → печатает репутацию, не двигая денег.

Корень доверия = `(адрес сплиттера в конфиге) × (неизменяемость сплиттера) × (неизменяемость индексатора)`. Три множителя, каждый проверяем публично.

**Корень признания №2 — пиннутые фабрики (шаг 6 карты проекта).** Признание события не меняется: только сплиттер. Меняется **атрибуция донора**: если `donor` события — эскроу, рождённый фабрикой из `factories[]` конфига, сеттлмент засчитывается его донору. Проверка — арифметика адреса через внешний крейт `crown-derive` (`crown-factory`, docs/factory-spec.md §4): у эскроу читаются `salt` и `donor` — поля аккаунта, владельцем которого обязана быть фабрика, — и адрес обязан пересчитаться как PDA из `(фабрика, salt)`. Раскладку заголовка фиксирует конвенция шейпов (factory-spec §2.1): `donor` на смещении 8..40, `salt` на 40..72 — у каждого шейпа, всегда; 32 байта, не пересчитывающиеся в адрес донора, не атрибутируют никогда. Реестра нет, доверия к событиям нет. Донор, не выводимый ни из одной фабрики — обычный кошелёк или чужая программа — засчитывается как есть. Транспортная ошибка чтения шва останавливает раунд (retry), а не атрибутирует неверно: книга неисправима. Несколько `Settled` одной транзакции (эскроу платит K получателям) засчитываются каждый по отдельности. `reduce/` этой ветки не видит, `REDUCE_VERSION` не менялся.

## 5. Индексатор: ингест, курсор, exactly-once

Ингест внутренний, запускают его двое: **пустой будильник и сторожевой таймер**.

`ingest_hint` — единственный `update` канистры: без аргументов, без ответа, без авторизации. Он умеет ровно одно — приблизить момент следующего чтения чейна; повлиять на то, *что* чтение найдёт, вызовом невозможно: признание, консенсус провайдеров и курсор не видят вызывающего. Клиент звонит в будильник после финализации своей транзакции — и запись появляется в книге через секунды. Ранний или повторный звонок не теряется и не множит расходы: внутри зазор (60 с) — сколько бы звонков ни пришло, канистра платит максимум за одно чтение на зазор; звонок внутри зазора взводит чтение на его границе, звонок во время идущего раунда добирается следующим.

Сторожевой таймер — гарантия полноты: книга обязана сойтись с чейном сама, даже если ни один клиент не позвонил (донат через сторонний клиент, умерший бот, закрытая вкладка). Его интервал — значение профиля `ingest_watchdog`: на devnet короткий (прогоны не ждут звонков), на mainnet длинный — сторож лишь добирает то, что никто не пнул.

Никакой вызывающий не может повлиять на **содержимое** состояния — write-канала в книгу по-прежнему не существует.

**Финальность решается коммитментом, не константой.** В конфиге нет `finality_depth`: `commitment = finalized`, всегда. Реорг до финальности → события просто нет в свёртке.

**Курсор — единственное состояние ингеста.** `getSignaturesForAddress(program_id, { until: last_sig, commitment: finalized })`, пагинация через `before`, обработка от старых к новым; затем `getTransaction(sig)` на каждую новую подпись. Курсор = последняя финализированная подпись.

Курсор монотонен + финальность необратима ⇒ exactly-once без множества «виденных». Память O(1).

**Solana: не полагайся на `logMessages`.** Программные логи усекаются при переполнении. Событие писать через event-CPI (оно попадает в inner instructions и не усекается).

SOL RPC при `encoding = base64` отдаёт структурированную мету целиком — `meta.innerInstructions` (со `stackHeight`), `meta.pre/postTokenBalances`, `meta.loadedAddresses`. Событие распознаётся как inner instruction пиннутой программы с тегом event-CPI; подделать его нельзя — инструкция требует подписи event-authority PDA, которую даёт только сама программа.

**Перекрёстная сверка — по исполненному переводу, не по дельтам балансов.** Дельты `pre/postTokenBalances` неоднозначны при совпадении ролей (donor == recipient), поэтому сверяется сама исполненная инструкция: `TransferChecked` непосредственно перед событием на той же глубине стека обязан переводить ровно `gross`, минт — USDC из конфига, authority — `donor` из события. Расходятся — сеттлмент не засчитывается, инкремент счётчика аномалий. Два независимых свидетельства — событие и перевод — должны совпасть.

## 6. Открытая книга

Все методы канистры — `query`. Бесплатно, permissionless. Читать может кто угодно, строить поверх может кто угодно; его чтение на книгу не влияет.

```
get_reputation(chain, donor, recipient) -> u128
get_cursor(chain)                      -> opt text
get_reduce_version()                   -> u32
get_certificate()                      -> (Certificate, BookRoot)
get_anomaly_count()                    -> u64        # счётчик аномалий §5
```

**Certified data.** Обычный `query`-ответ подписан одной репликой и потому третьему лицу не доказывает ничего. Индексатор держит меркл-корень книги в `set_certified_data`, а `query` отдаёт `data_certificate()`. Ответ проверяется против root key NNS без доверия оператору.

Именно это заменяет отдельный проект-verifier. А кому мало сертификата — берёт крейт `reduce` (нулевые зависимости, публичный, версионированный), скармливает ему те же публичные `Settled` и получает ту же книгу. «Пересчитай сам» — это `cargo run`, а не репозиторий.

## 7. Конфиг-поверхность

Индексатор знает про сеть ровно шесть вещей:

```toml
[[chain]]
id       = "solana-devnet"        # внутренний ChainId ядра (кластер)
source   = "Default:Devnet"       # вариант SOL RPC-канистры; на mainnet только Default:*
consensus = "3-of-5"              # стратегия агрегации провайдеров
splitter = "<program id>"         # корень признания
usdc     = "<mint>"               # для перекрёстной сверки переводов
factories = ["<program id>"]      # корни признания №2, ноль или больше (§4)
```

Плюс один ключ уровня профиля — `ingest_watchdog`, секунды сторожевого интервала ингеста (§5): не про сеть, а про темп страховки; testnet — 60, mainnet — 1800.

**Чего в конфиге индексатора НЕТ:** чьи-либо комиссии, `min-donation`, RPC-URL, API-ключи, глубина финальности, схема подписи.

Единственный параметр **деплоя контракта** — минт USDC (`contracts/*/deploy.toml`). Комиссии игр — параметры их эскроу (`crown-factory`); индексатор их не читает и не проверяет. Он не может быть обманут неверным значением комиссии, потому что не имеет мнения о комиссиях. Он доверяет адресу.

devnet → mainnet = смена профиля. Ноль изменений кода.

## 8. Сеть и вход средств

Сеть одна — **Solana**. Выбор не случаен, а вынужден требованиями ядра:

- **Донат должен стоить центы, а книга — обновляться на глазах.** Финальность Solana — секунды; репутация появляется в книге в реальном времени.
- **Чтение чейна — только `Default`-источники RPC-канистры под NNS.** Всё, что требует своего URL (`Custom`), на mainnet запрещено — иначе оператор становится единственным источником истины и вся конструкция бессмысленна. SOL RPC покрывает `Default`-провайдерами и Mainnet, и **Devnet** — есть где честно тестировать.
- `Custom` разрешён **только в testnet-профиле** (локальные прогоны). Линт в CI: mainnet-профиль, содержащий `Custom`, роняет сборку.

SOL RPC-канистра сама обращается к третьесторонним провайдерам (Helius, Ankr, dRPC, PublicNode и др.) и агрегирует ≥3 ответов. «Без посторонних RPC» невозможно; достижимо «без **своих** RPC и своих ключей, с консенсусом под управлением NNS». Это и есть требование.

**Вход средств из других сетей — не задача ядра.** Донор, чей USDC лежит в другой сети, пополняет свой Solana-кошелёк через CCTP (нативный перенос USDC от самого Circle: burn там → mint нативного минта здесь) — это слой `crown-app`, вне доверенного контура, подключается и меняется в любой момент, хоть после заморозки. Донат всегда подписывает солановский кошелёк донора; книга про мост не знает и не узнает.

## 9. Неизменяемость

Сплиттер immutable с момента деплоя — иначе гарантии §3 ничего не стоят.

Индексатор замораживается (blackhole) на S4. Обоснование не «чтобы не украли» — красть нечего, — а **честность закона**: замороженный `reduce` нельзя переписать задним числом. Это единственная гарантия, потому что независимого verifier-сервиса у нас нет.

Цена честная: **баг в замороженном индексаторе неисправим.** Митигация не в апгрейде, а в природе книги: она derived. Битый индексатор выбрасывается и заменяется новым, который пересчитывает всё из чейна с нуля. Активом является пара (адрес сплиттера, история чейна), а не канистра. Поэтому заморозка дёшева.

Требование до заморозки: **reproducible build**, хэш wasm опубликован. Иначе «заморожен» ничего не значит.

## 10. Принятые ограничения

Не латать, а принять.

1. **Самосибил возможен и дёшев.** Кошельки не связываются, можно донатить себе с многих; деньги возвращаются себе же, эмиссия стоит только сетевых сборов. Книга — реестр фактов платежей, а не механизм сопротивления сибилу. Любой потребитель, использующий её как вес (голосования игр, гейты), обязан приносить собственную защиту; ядро её не обещает.
2. **Репутация — плутократия по построению.** В money-backed системе это свойство. Разбавить «заработанной» компонентой ядро не может — репутация тождественна деньгам.
3. **Репутация локальна кластеру.** Книги devnet и mainnet — разные вселенные; между ними ничего не переносится. Склейка личностей поверх кошельков — вне ядра и никогда не пишет в книгу.
4. **Permissionless-чтение и будильник жгут циклы.** Query бесплатны для читателя и оплачиваются канистрой (reverse gas); `ingest_hint` бесправен, и спам им — тоже статья расхода. У спама встроенный потолок: зазор (§5) ограничивает платные чтения одним на 60 секунд — непрерывный злонамеренный спам возвращает канистре ровно счёт постоянного минутного опроса, и только пока длится; сам вход вызова стоит копейки. Митигация остального — rate limit на boundary nodes и бюджет; бюджет — на S4.
5. **Доверие к RPC-канистре.** Мы доверяем NNS-управляемому коду и его набору провайдеров. Это сокращение доверия, а не его отсутствие.
6. **Ядро требует дешёвой сети.** Донат должен стоить центы — Solana этому удовлетворяет.

## 11. Ядро бесплатно; монетизация — в играх

- Сплиттер комиссии не берёт: получатель получает донат целиком. Ядро не зарабатывает и не имеет казны.
- Чтение книги бесплатно и permissionless.
- Монетизация живёт в играх: создатель игры задаёт свою комиссию и свой кошелёк в параметрах рождения эскроу (`crown-factory`). Комиссия снимается эскроу прямым переводом **до** сплиттера и в книгу не попадает: репутация — это то, что дошло до получателя, а не то, что заработала игра. Игра enforce'ит комиссию сама — подпись исхода, отпирающая расчёт, выдаётся только эскроу, рождённому с её параметрами.
- Циклы канистры — операционный расход оператора: outcalls ингеста + permissionless-query. Источник финансирования — вне ядра (например, комиссии собственных игр платформы); механизм пополнения — S4.

Комиссии конкурируют: у каждой игры своя, донор видит её до подписи транзакции, прямой донат мимо игр не стоит ничего.
