# Crown — Карта проекта

Как устроен продукт целиком: репозитории, участники, потоки, порядок сборки.

Стадия ядра — `build-plan.md`. Отложенное и его триггеры — `post-core.md`. Границы и тесты — `standards.md`. Ядро — `core-spec.md`.

Ничего не деплоится на mainnet до готовности продукта. До этого момента всё живёт локально и на devnet, передеплой свободен.

---

## 1. Что происходит

Донор платит получателю в USDC — получатель получает сумму целиком. Факт платежа попадает в открытую книгу репутации: `(chain, donor, recipient) → Σ gross`.

Два пути платежа:

- **прямой** — донор зовёт сплиттер сам; никакой комиссии;
- **через игру** — деньги лежат в эскроу, игра решает исход, эскроу зовёт сплиттер. Комиссию и кошелёк комиссии задаёт создатель игры; они вшиты в эскроу при рождении, снимаются до сплиттера и в книгу не попадают.

Оба пути приводят в одну книгу. Второй существует только когда есть фабрика. Монетизация платформы — комиссии её собственных игр; ядро не зарабатывает.

## 2. Репозитории

| Репозиторий | Что внутри | Ключи | Деньги | После mainnet |
|---|---|---|---|---|
| `crown-core` | `reduce/`, `index/`, `contracts/`, `config/` | нет | нет | заморожен |
| `crown-factory` | `Factory`, `Escrow` | нет | нет | иммутабельна |
| `crown-games/<имя>` | логика игры, резолв | свои | нет | свободен |
| `crown-app` | `web/`, `api/`, `moderation/`, `mirror/` | есть | нет | свободен |
| `crown-ops` | деплой, циклы, воспроизводимая сборка | все | операционные | свободен |

`crown-factory` — отдельный репозиторий: байткод `Escrow` вшивается в байткод `Factory` при компиляции.

## 3. Доверенный контур

Два адреса на сеть, оба в `config/` индексатора:

```toml
splitter  = "<адрес>"     # корень признания №1
factories = ["<адрес>"]   # корни признания №2, список: новая форма = строка
```

Всё, что не выводится из этих двух адресов, для книги не существует.

| | Пиннут | Держит деньги | Может врать |
|---|---|---|---|
| Сплиттер | да | нет | нет |
| Фабрика | да | нет | нет |
| Эскроу | нет — адрес вычисляется из фабрики | да, до расчёта | нет |
| Игра | нет | нет | да |
| `crown-app` | нет | нет | да |
| `crown-ops` | нет | да, операционные | да |

Между сплиттером, фабрикой и индексатором нет каналов. Никто из них не знает о существовании остальных.

## 4. Потоки

**Прямой донат.**

```
Донор ──donate(recipient, gross)──► СПЛИТТЕР ──► получатель  (gross целиком)

                        emit Settled(donor = донор, recipient, gross)
```

**Донат через игру с холдом.**

```
игра ──createEscrow(donor, recipient, gross, deadline)──► ФАБРИКА
                                                            │ рождает
                                                            ▼
Донор ──USDC──────────────────────────────────────────► ЭСКРОУ

     ┌── settle()  ── зовёт кто угодно, после решения игры
     │      ЭСКРОУ ──комиссия──► кошелёк игры   (прямой перевод, без события)
     │      ЭСКРОУ ──donate────► СПЛИТТЕР ──► получатель  (остаток целиком)
     │      emit Settled(donor = адрес эскроу, ...)
     │
     └── refund()  ── зовёт кто угодно, после deadline
            ЭСКРОУ ──► донор, все 100%.  Settled нет, репутации нет, комиссии нет.
```

Игра держит только кнопку `settle()`. Донор, получатель, сумма, дедлайн, комиссия игры и её кошелёк вшиты в эскроу при рождении. Комиссия берётся только при расчёте: игра зарабатывает, только когда донат состоялся.

**Вход средств из других сетей.** Не поток ядра: донор пополняет свой Solana-кошелёк через CCTP (нативный USDC от Circle) на уровне `crown-app`, дальше — обычный донат его подписью.

**Ингест.**

```
Settled в блоке ──► SOL RPC-канистра (NNS, ≥3 провайдера) ──► crown-index ──► книга
                                                                            │
                                                               query, бесплатно,
                                                               сертифицировано
                                                                            ▼
                                                                      кто угодно
```

## 5. Признание

Внутри `index/`. Признаётся только `Settled`, эмитнутый пиннутым сплиттером (как и всегда); ветка — про **атрибуцию донора** этого события:

```
donor выводится из фабрики ∈ config.factories → прочитать donor() у donor, засчитать донору
иначе                                          → засчитать donor как есть
```

Вывод — арифметика: пересчёт PDA из `[b"escrow", salt]` (данные аккаунта обязаны принадлежать фабрике). Реестра нет; шов donor-а — два поля аккаунта: `salt` и `donor`.

`reduce/` этой ветки не видит. `REDUCE_VERSION` от появления фабрики не меняется.

---

## 6. Книга

Четыре точечных `query`, все сертифицированные:

```
get_reputation(chain, donor, recipient)
get_cursor(chain)
get_reduce_version()
get_certificate()
```

Книга **не умеет**: перечисление, время, порядок, расход репутации, кросс-чейн и кросс-получательные агрегаты.

Всё это делает `crown-app/mirror/` — собственный индексатор на публичном крейте `crown-reduce`, читающий те же события из цепочки. Он перечисляет, канистра доказывает.

## 7. Порядок сборки

| # | Что | Репозиторий | Трогает ядро |
|---|---|---|---|
| 1 | Сплиттер + индексатор, devnet | `crown-core` | — |
| 2 | Минимальный фронт: донат и чтение книги | `crown-app/web` | нет |
| 3 | Игры класса **A** (2–3 штуки) | `crown-games/*` | нет |
| 4 | Первая игра класса **B**: её спека = ТЗ на `Escrow` | `crown-games/*` | нет |
| 5 | Фабрика и шаблон под это ТЗ | `crown-factory` | нет |
| 6 | Ветка `is_child_of` + поле `factory` в конфиге | `crown-core/index` | **один раз** |
| 7 | Остальные игры класса B | `crown-games/*` | нет |
| 8 | Полноценный фронт | `crown-app/web` | нет |
| 9 | Модерация, операторство, `mirror` | `crown-app` | нет |
| 10 | Mainnet: деплой всего, финализация authority, blackhole | `crown-ops` | — |

Классы игр:

- **A** — деньги уходят получателю сразу и безвозвратно, целиком. Ноль контрактов — и потому ноль комиссии: enforce'ить её нечем. Фабрика не нужна.
- **B** — деньги лежат в эскроу, репутация начисляется донору при расчёте, комиссия игры — с эскроу при расчёте. Требует фабрику.

Шаг 5 невозможен раньше шага 4: форма `Escrow` определяется требованиями первой игры класса B.

Шаг 6 — единственное изменение в `crown-core` за всю жизнь проекта. Шаги 1–9 идут на devnet, передеплой свободен. Шаг 10 — однократный и необратимый.

## 8. Правила

- Ключи не лежат в репозиториях доверенного контура. `crown-core` и `crown-factory` клонируются и собираются кем угодно в тот же хэш.
- Write-канала «наружу → внутрь» не существует. Единственный вход в книгу — транзакция через сплиттер, оплаченная реальными деньгами.
- Игры произвольны и никем не одобряются. Ограничение наложено на кассу: репутация донору начисляется только через эскроу от пиннутой фабрики.
- У каждой строки есть текущий потребитель. Пока пункт стоит в `post-core.md` — под него нет ни кода, ни хука, ни поля конфига.
