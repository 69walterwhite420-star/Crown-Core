# Crown — Post-core

Всё, чего **нет** в этом репозитории, и почему. Каждая строка — сознательно отложенный кусок с известным швом подключения. Ядро от их отсутствия не страдает: ни один из них не находится в доверенном контуре.

Правило: **пока строка здесь — в `crown-core` не должно быть ни строчки кода, ни хука, ни фича-флага под неё.** «Подготовить место» — это и есть спекулятивная обобщённость, которую запрещает `standards.md`.

Реализовал — заводи отдельный репозиторий, а строку отсюда удаляй.

## Решения, а не код

| Что | Почему не сейчас | Триггер |
|---|---|---|
| **Оператор-канистра** | гибрид «инспектируемый стек vs простой внешний бэкенд». Не касается ядра | когда единый инспектируемый стек ценнее простоты |
| **Юрисдикция инкорпорации и допуск юзеров из США** | мастер-решение, от которого расходится вся регуляторная карта | до mainnet (S4) |
| **Юр. консультация до хостинга чужого доллара** | ядро закрывает построением: гэмблинга нет (нет приза), денежного трансмиттинга нет (нет кастодии), Howey нет (репутация непередаваема и не растёт от игр). Остаточно: AML/санкции без KYC, посредническая ответственность за офчейн-текст | до mainnet (S4) |
| **Механизм пополнения циклов** | конвертация операционных средств платформы (у ядра казны нет; источник — например, комиссии её собственных игр) в cycles заблэкхоленной канистры | **до** снятия контроллера на S4. Порядок критичен: обратного пути нет |
| **Склейка личностей поверх кошельков** | репутация per-wallet (core-spec §2). Несколько кошельков одного человека — несколько записей | отображение может показывать их рядом. **Книга об этом не узнаёт никогда** |
| **Вход средств из других сетей** | это пополнение кошелька донора, а не донат: CCTP (нативный перенос USDC от Circle) кладёт USDC на солановский адрес донора, дальше обычный `donate` его подписью | слой `crown-app`, вне контура; подключается в любой момент, хоть после заморозки |
| **Сплиттеры на других сетях** | второй криптостек, свои RPC, медленная финальность — при CCTP-входе не покупают ничего | только как осознанное расширение доверенного контура, решение до заморозки |

## Доверенный контур после заморозки

Фабрики уже в контуре: признание события — пиннутый сплиттер, атрибуция донора — пиннутые фабрики (`factories[]`, core-spec §4). Пока индексатор не заморожен, новая фабрика — это строка конфига и передеплой.

После заморозки список вшит навсегда. Новая фабрика (или новый сплиттер) = новая канистра, которая пересчитывает всю историю из чейна с нуля. Это нормально и предусмотрено: книга derived, активом является пара (адрес сплиттера, история чейна), а не канистра.

Именно поэтому в ядре нет «места под будущие фабрики» и нет мнения о комиссиях игр: либо адрес в конфиге со всеми доказательствами, либо его для книги не существует вовсе.
